## Problem 1 — The sidebar didn't go anywhere, your viewport is just under 768px

Diagnosis: `src/components/app-sidebar.tsx` line 166 — the `<aside>` uses `hidden md:flex`. Tailwind's `md:` breakpoint = 768px. Your current viewport is **740px**. Below 768, the sidebar is hidden and there's no trigger to bring it back. That's the bug. Nothing was weakened — the 7 groups, all items, KTIs, everything is still in the file exactly where it was.

**Fix (per your "just fix the disappearance" answer):**

1. Add a hamburger button to `src/components/app-topbar.tsx` (left of the search input), visible only `md:hidden`. Clicking it opens the existing sidebar content inside a shadcn `<Sheet>` drawer from the left.
2. Refactor `src/components/app-sidebar.tsx` so the nav body is a separate `<SidebarNav>` component reused by both the desktop `<aside>` and the mobile `<Sheet>`. Zero content changes — same `GROUPS` array, same labels, same routes, same KTI entry under Library.
3. Drawer auto-closes on route change (TanStack `useLocation` effect).

That's it. No content audit, no item changes. After this: viewport ≥768 = sidebar pinned (today's behavior), viewport <768 = hamburger opens the same nav as a drawer.

## Problem 2 — SweetScan is too narrow. Promote it to a first-class surface.

Today, SweetScan only mounts inside one Domain detail page. That's why it feels invisible. You said you're "not sure where it should live" — let me reframe it before answering, because the framing changes the answer.

### What SweetScan actually IS

SweetScan is the **outside-in intelligence layer**. Two distinct jobs it does today, one job it should also do:

| Job | What it pulls in | What it writes |
|---|---|---|
| **Rubric scanner** | External best-practice signals for a Domain | Excellence-Rubric checklist proposals |
| **Forward radar (KTIs)** | Threshold-watched signals per Domain or Relationship | KTI scans + fires → spawn Tasks/bot alerts/Flightdeck flags |
| **(missing) Inbound capture** | Articles, podcasts, posts, transcripts dropped via Capture | Tagged signals routed to whichever Domain/Relationship/KTI they touch |

It's not a domain-page widget. It's the **eyes and ears** of the OS. Right now it only has eyes for one Domain at a time.

### Proposed home: `/sweetscan` as a top-level route in **Think**

Sidebar entry: **Think → SweetScan** (sits between BizzyBots and Decisions). One page, three tabs:

1. **Forward radar** — global KtiPanel: every active KTI across every Domain and every Relationship, sorted fired→trending→flat. Filters: scope (universal/per-client), domain, status. This is the "what's lighting up right now" view.
2. **Rubric scanner** — pick a Domain (dropdown), run a scan, review proposed checklist items. Same UI as today's `SignalScannerConfig` — just no longer trapped on one Domain page.
3. **Signal inbox** — list of recent scan rows (rubric proposals + KTI scans), each linkable to the Domain/Relationship/KTI it touched. This becomes the audit trail of what SweetScan saw and what it did with it.

### Embedded surfaces (where SweetScan also shows up, but as a *view*, not a home)

These already exist — just confirming the pattern stays:

- **Domain detail** — keeps `<SignalScannerConfig>` (Forward radar tab + Rubric scanner tab) scoped to that domain. Unchanged.
- **Relationship detail** — keeps `<KtiPanel relationshipId={id} />`. Unchanged.
- **Today** — add a compact **"Fired KTIs"** strip at the top of `/today` showing fires from the last 24h across all clients. One line each, click → KTI detail. This is the daily morning glance.
- **Flightdeck** — add a **"Fired" column** with count of fired KTIs per relationship. Click → that relationship's radar.

### Why this is the right home

SweetScan watches the outside world; that's a Think-layer job, not a Library catalog or a Today action. Putting it in Think makes the verb match ("what's the world telling us?") and gives it room to grow without polluting any one Domain or Relationship page.

## Problem 3 — Story Trail: add Operator detail + a Master Story Trail

You want both. Both are read-only composes of existing data; no schema changes.

### A. Operator detail Story Trail

Mount `<StoryTrail subjectKind="operator" subjectId={id} />` on `/operators/$id`. The component gets a new branch:

- **Sparks** generated by this operator (operator_id on sparks, if set)
- **Component outputs** this operator produced (created_by_operator_id on component_outputs)
- **Decisions** logged by this operator
- **Workflow step runs** this operator completed (audit log filter)

Renders chronologically, same chapter format as the other four kinds.

### B. Master Story Trail at `/today` (top of page, collapsible)

A "Story" surface that shows the **last 20 meaningful beats across the entire workspace** — every relationship, every quest, every component. Same beat types (spark · output · decision · audit), each tagged with its origin (which client, which quest). Filters: kind (spark/output/decision/audit), scope (mine/everyone), last 24h / 7d / 30d.

This is the daily "what happened?" glance. Sits above Today's planner so the first thing you see is recent narrative, not just upcoming work.

Implementation: extend `story-trail.tsx` with a fifth subjectKind `"workspace"` that drops the `subjectId` filter and pulls the 20 most recent beats across all four sources. New component `<MasterStoryTrail />` wraps it with the filter strip.
