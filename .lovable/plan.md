## 1. Spark vs Task — the canonical distinction

Per your docs (`01-canonical-system-truths`, `03-extensibility-hooks`):

| | **Spark** | **Task** |
|---|---|---|
| **Origin** | System-generated only | Human, system, OR operator |
| **Lives inside** | A Quest → Journey → Mission (client-side SweetSync) | A Project, or standalone (operator-side execution) |
| **Purpose** | Atomic interaction that *advances a Component* (Question, Creation, Definition, Decision, Reflection, Action) | Atomic executable work to *get something done* |
| **Audience** | Client (SweetSync) | Operator / team / delegate |
| **Outputs** | `captured_answer`, advances `affected_components` | `output_link`, `deliverable_specific` |
| **Lifecycle** | progression_state enum (Pre-filled → Provisionally Satisfied → Confirmed Complete…) | status (To Do / Doing / Done) |

### What I'll change

**Schema (additive, lossless):**
- Add `sparks.generated_by_kind` enum (`'system' | 'agent' | 'workflow'`) — **no `'human'` value**, enforced by CHECK constraint. Default `'system'`.
- Add `sparks.generator_operator_id uuid → operators(id)` — which agent/workflow created it (nullable for pure system).
- Add `sparks.origin_event` text — short label ("Mirror session inferred", "Domain Assessment gap", "Curator agent run") so provenance is visible.
- Tasks already has `source` (`manual | ai | workflow | import`) — keep, but rename UI label to "**Created by**" everywhere.

**RLS / triggers:**
- Trigger `trg_spark_block_human_create` raises if INSERT comes with `generated_by_kind = 'human'` or via a non-service-role direct user insert. Sparks can only be created by edge functions (service role) or via the proposal queue.
- The `/sparks` UI loses the "+ New" button. Replaces it with a banner: **"Sparks are generated by the system. Create a Quest or run a Curator to spawn Sparks."**

**UI clarity:**
- Spark list page header: explainer card — "**Sparks are atomic interactions the system generates to advance a Component.** They show up here for review; you confirm, edit, or skip."
- Each Spark card shows a provenance chip: `🤖 system` / `🤖 {agent name}` / `⚙️ {workflow name}` + the `origin_event`.
- Tasks list keeps its existing "+ New Task" and shows a `Created by` chip: `👤 you` / `🤖 agent` / `⚙️ workflow` / `📥 import`.

## 2. Organize the system per the canonical docs

The docs split things into **four layers**. The sidebar should mirror this exactly. Today the sidebar groups are loose and don't match the canon.

### Proposed sidebar IA (matches docs 1:1)

```text
TODAY                          ← operational write surface
  Today · Capture · Queue · Planner · Calendar · My Tasks

OPERATE (operator side)        ← SweetCycle delivery
  Pipeline · Flightdeck · SweetCycle · Sessions · Measures
  Engagement Plans · Campaigns · Delegation · Decisions · Documents

LIBRARY (definitions)          ← what CAN be done
  Workflows · Session Templates · Playbooks · Personas · Components
  Journeys · Quests · Outcomes

SWEETSYNC (client side, system-generated) ← future-Phase-3, scaffolded now
  Missions · Sparks · Domain Assessments

PEOPLE & RELATIONSHIPS
  Relationships · People · Operators · Projects

TAXONOMIES
  Domains · Tenets · Lenses (BizzyBots)

SETTINGS
  Excellence Rubric · BizzyBot Prompts · Team
```

**Key relocations:**
- **Sparks** moves from wherever it is today to **SweetSync** group (it's client-facing, system-generated — per canon).
- **Quests, Journeys, Missions** also move to SweetSync (transformation chain).
- **Components** stays in **Library** (definitions side) — they're the building blocks.
- **Outcomes** moves to **Library** (definition catalog of measurable result types).
- **Tasks** stays accessible via Today → My Tasks and inside each Project.

Each group gets a one-line caption in the sidebar header explaining what it contains (the docs language).

### Page-purpose lines updated to match canon

Use the exact docs language on every PageHeader:
- **Sparks** — "Atomic interactions the system generates — Question, Creation, Definition, Decision, Reflection, Action — to advance a Component."
- **Tasks** — "Atomic executable work. Created by you, an agent, or a workflow. Has a clear deliverable."
- **Quests** — "Structured work units that group Sparks and advance one or more Components, producing a Deliverable."
- **Journeys** — "Capability areas that contain Quests and Components."
- **Missions** — "The overarching transformation goal. One active per client. Activates Journeys."
- **Components** — "Concrete building blocks being matured (e.g. 'Service Tiers'). Have maturity levels. Advanced through Quests, Sessions, and Tasks."
- **Workflows** — "The underlying things being made real. Stored, versioned, reusable. Capabilities are derived from these."
- **Outcomes** — "Measurable business results. Six types: Time Saved, Revenue Increased, Efficiency Gained, Satisfaction Improved, Cost Reduced, Quality Improved."

## 3. Reinforce the canonical distinctions in the data

Two small but important schema reinforcements pulled straight from the docs:

**a) `state_of_the_thing` per Workflow per Relationship** — a join `workflow_relationship_state` table:
```sql
CREATE TABLE workflow_relationship_state (
  workflow_id uuid, relationship_id uuid,
  state_of_the_thing text CHECK (state_of_the_thing IN
    ('Identified','Defined','Designed','Built','Delivered','Adopted','Sustained')),
  source_of_advancement source_of_advancement,
  last_updated, notes,
  PRIMARY KEY (workflow_id, relationship_id)
);
```

**b) `progression_state` enum extension** — current enum is partial; add the missing canonical values: `'Pre-filled'`, `'Provisionally Satisfied'`, `'Completed by you'`, `'Completed with Liz'`, `'Completed for you'`, `'Confirmed Complete'`, `'Reopened'`, `'Superseded'`. Sparks use the full enum; Tasks keep simple status.

Both are pulled directly from `01-canonical-system-truths` §13 and §15.
